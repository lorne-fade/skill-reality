The de-risk spike

One to two weeks, a throwaway harness, and five measurements that decide the whole architecture. Meta ships the camera, audio, and display access — but not the numbers. This spike gets the numbers before a line of product code is written.

1–2 weeks Throwaway harness Gates Phase 1

What this spike must answer

Five questions, in priority order. The first two can end the "continuous real-time overlay" dream on their own — run them first. Everything downstream (model, UX, battery strategy) is designed around these answers, so guessing them is the most expensive mistake available.

Track A · days 1–2

◫ Mock Device Kit

Stand up the SDK against Meta's simulated device before hardware is in hand. Learn the real API surface — how you subscribe to frames, how you push to the display, what permissions fire — and build the logging harness.

De-risks integration so the physical-device days are spent measuring, not fighting the build.
Track B · days 3–7

◉ Physical unit

The same harness on a real Meta Ray-Ban Display + paired phone. This is where the numbers come from — cadence, latency, display behavior, battery — under conditions the Mock Kit can't fake.

Mock fidelity is itself an unknown (M5); trust only the physical numbers for go/no-go.

  • Developer Mode on (tap version 5× in App Info, then toggle)
  • Glasses paired to the Meta AI app; joined an invite-only release channel
  • Confirm the tester cap and any approval lead time

Hardware & tooling

  • Meta Ray-Ban Display unit, fully charged + a spare if possible
  • Paired iPhone (iOS 15.2+) or Android 10+ phone
  • Xcode 14+ or Android Studio Flamingo+; DAT SDK from GitHub
  • Firmware + Meta AI app versions aligned to the SDK's dependency table

The five measurements

method · target · gate
M1How fast can we get frames off the glasses?
Method

Subscribe to the camera stream; log arrival timestamps for 60s. Compute mean & 95th-percentile FPS, jitter, resolution, and pixel format. Note whether it's a continuous push-stream or a pull-per-capture.

Target

≥ 5 fps sustained (≥ 15 ideal), short side ≥ 640 px so small objects survive. Record format — JPEG vs raw YUV changes the pipeline.

No-go

Under ~2 fps means "continuous real-time" is off the table — reframe as a periodic scan, and design the UX around that from the start.

M2What's the glass-to-alert latency, end to end?
Method

Timestamp three legs: frame captured → frame ready on phone → alert visible on the HUD. Film the display leg with a high-frame-rate phone camera against an on-screen millisecond clock.

Target

< 1.5 s capture-to-alert for a usable warning; < 0.7 s excellent. Budget ~30–60 ms for on-device inference inside that.

No-go

Over ~3 s and alerts are advisory, not "avoid-it-now" — which points to on-demand scanning rather than always-on watching.

M3What can the HUD actually draw?
Method

Attempt three outputs in order: (a) a text card, (b) icon + text alert, (c) a positioned box / free graphic. Record which succeed, plus max refresh rate, dwell time, color depth, and safe-area limits.

Target

Classify the surface: notification-only, card, or free-draw overlay. This single answer decides whether boxes live on the glass or in phone/review.

Constraint

If it's card-only (the likely case), the on-glasses UX is a glanceable alert; the bounding-box view moves to the phone companion and post-shift review.

M4How long does continuous capture last?
Method

From full charge, run capture plus a stub inference load at target FPS continuously. Log battery % per minute on both glasses and phone; note any thermal throttling or heat warnings.

Target

≥ 45–60 min continuous on the glasses for demo viability; a shift-long tool likely needs duty-cycling regardless.

No-go

Under ~20 min means always-on is impractical — design capture to duty-cycle: wake on gesture or context, sleep otherwise.

M5What do access, privacy, and the Mock Kit really impose?
Method

Log the approval lead time and tester cap. Observe the capture-indicator LED and any consent prompt behavior. Diff one Mock Device Kit run against the physical unit to gauge fidelity.

Target

A known tester ceiling, a documented capture-consent behavior, and a sense of how far the Mock Kit can be trusted for future dev without hardware.

Watch

Always-on POV camera carries real consent obligations — start the data-handling stance now, not at pilot.

Day by day

buffer the front for access approval
Days 1–2
Access + Mock Kit + harness. Clear the checklist, integrate the SDK against the Mock Device Kit, and build the throwaway logging harness (timestamps to CSV, a few toggles).
Days 3–4
M1 + M2 on hardware. Camera cadence and end-to-end latency on a physical unit — the two answers that can end the continuous-overlay path.
Days 5–6
M3 + M4. Probe what the display renders; run the battery + thermal soak with a stub inference load.
Day 7
M5 + edges. Access mechanics, capture-consent behavior, Mock-vs-real fidelity diff.
Days 8–9
Analyze + decide. Fill the results table, run the decision matrix, write the one-page findings memo, and lock the Phase 1 architecture variant.

Decision matrix

outcome → what you build in Phase 1

The spike doesn't pass or fail — it selects a design. Each outcome routes to a different, still-valuable product. The only real failure is not knowing which one you're in before you commit.
