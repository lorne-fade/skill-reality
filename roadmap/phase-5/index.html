Three tracks, run in parallel

▣ Fleet

The operator's view — what a safety manager buys.

  • Dashboard — alerts, hazard heatmap by zone/shift, trend lines; reads the Phase 3 event schema as-is
  • Audit trail — every alert and every suppression, with the policy version that produced it
  • OTA updates — signed model + policy bundles, staged rollout, one-tap rollback
  • Device management — enrollment, health, battery, per-site policy assignment

◆ Platform

The engineering debt the slice deliberately deferred.

  • Crash + telemetry — anonymous, opt-in, no frame content ever
  • Battery duty-cycling — productize whatever mode Phase 0/4 forced
  • Android port — TFLite path, reusing the pipeline seams
  • Per-site policy config — the SME table becomes deployable data, versioned per site
  • Degraded modes — sensor-only fallback (the original Hazard Watch!) when camera or phone drops

◈ Compliance

What lets procurement and legal say yes.

  • Data-handling whitepaper — on-device inference, what leaves, retention, deletion
  • DPA + consent templates — the pilot pack, productized per jurisdiction
  • No-surveillance commitment — published, versioned, contractual
  • Regional review — GDPR / works-council playbook from the pilot experience
The repeatability test

Adding a hazard class must be a process, not a project

The Tier 2 and Tier 3 classes — and every site-specific hazard a customer asks for — go through one standing pipeline. If adding a class requires an architecture conversation, Phase 5 isn't done. The gate: a new class ships end-to-end in under two weeks without touching pipeline code.

Collect (flywheel + synthetic)→Label vs. ontology→Train + eval vs. recall floor→SME writes the policy row→OTA staged rollout→Field-monitor 2 weeks

The Meta publish path

distribution is the one clock we don't control
StageWhat it means for us
Now — invite-only channelsDistribute to our own org and pilot testers via release channels. Every pilot and early customer runs this way — it's a deployment constraint to design around (named testers, per-channel builds), not a blocker.
Select partnersMeta is publishing some partner integrations ahead of GA. The pilot report and a reference customer are exactly the application material — apply as soon as both exist.
2026 — general availabilityPublic listing opens to a broader audience. Target state: land there with a hardened fleet product, N pilot-proven sites, and the compliance pack — not a prototype scrambling to productize.

Strategy note: because enterprise safety sells B2B through pilots anyway, the gated period costs us little — invite-only channels cover paid deployments with named crews. GA matters most for scale and for procurement teams that require "published app" status.

Definition of "product"

the checklist that ends the roadmap
  • ✓A new site deploys in days — templated consent pack, policy config, device enrollment, no engineers on site.
  • ✓A new hazard class ships in weeks — through the class factory, gated by recall floor + SME policy row.
  • ✓Models update over the air — staged, signed, reversible.
  • ✓The safety manager self-serves — dashboard + audit trail answer their questions without us.
  • ✓Compliance is a pack, not a negotiation — whitepaper, DPA, no-surveillance commitment, regional playbook.
  • ✓Degraded gracefully — camera loss falls back to the sensor monitor; nothing fails silent.
  • ✓Two platforms, one pass bar — iOS and Android hitting identical protocol results.
  • ✓Publish-ready — the Meta application submitted with proof attached.
The through-line, closed. The roadmap opened with a web demo that simulated a camera it couldn't have. It closes with a fleet product where the same brand system, the same hazard vocabulary, the same alert language, and the same event schema run from the first sensor demo to the operator dashboard — and where every deployed pair of glasses feeds the data flywheel that makes the next model better. Phases 0–4 bought certainty; Phase 5 buys repeatability. When M5 is submitted, the roadmap's job is done and the product's job begins.

Milestone scope assumes the Phase 4 pilot passed its bar; a partial pass reorders M1–M3 around the classes that fell short. Meta's partner-selection criteria, GA timing, and publishing requirements are Meta's to define — M5 is written against their current public preview terms (invite-only channels now, select partners ahead of a 2026 GA) and should be re-checked against their docs at execution time. Pricing/packaging is deliberately out of scope here; it feeds from the pilot report and belongs to the commercial plan.
