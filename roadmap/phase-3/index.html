The judgment layer —
detections aren't alerts

Phase 2 hands us a model that sees hazards. Phase 3 decides what deserves the worker's attention — because a warning system that interrupts too often gets muted, and a muted system catches nothing. This phase builds the policy between detection and alert, and the calm on-glasses UX it drives.

The design constraint everything follows from

Worker attention is a budget — spend it like money

A frontline worker mid-task will tolerate a handful of interruptions per shift before the glasses come off or the audio gets ignored. So the judgment layer is an economist, not a megaphone: every alert must clear a bar of severity × confidence × novelty, and everything that doesn't clear it gets logged silently instead. The log misses nothing; the worker hears only what matters. "More alerts" is never the answer — better-spent alerts is.

Five judgments between detection and alert

replaces Phase 1's single rule at stage 4
1

Persist per-class K-of-N · IoU tracking

Phase 1's 3-of-5 rule, generalized: each class gets its own confirmation window tuned to its physics — a spill is static (wide window, high certainty), a person entering frame is fleeting (tight window, faster trigger). Simple IoU tracking gives each detection an identity across frames, so the system knows this spill from a second one.

2

Score severity policy per class

The Phase 2 taxonomy's severity (critical / high / medium) is the base score, adjusted by confidence and proximity cues (bigger box ≈ closer ≈ more urgent). The safety SME owns this table — engineering implements it, and every change to it is versioned and logged.

SME-owned policy tableversionedproximity weighting
3

Arbitrate one alert at a time

A 600×600 glass shows one thing well and two things badly. When multiple hazards are live, the highest severity wins the display, full stop. Lower ones wait — and if they resolve or go stale while waiting, they're dropped to the log, not queued for a nag.

single-slot displayseverity preemptionstale alerts drop, never queue
4

Suppress acknowledgment · cooldowns · novelty

Acknowledged hazards stay suppressed for that specific tracked hazard — acknowledging one spill doesn't silence spills as a category. Re-alert only on real change: severity escalates, the hazard moves closer, or the ack expires. This is the "don't cry wolf" machinery, spelled out below.

per-hazard ackescalation re-armscategory never muted by one ack
5

Route severity → modality matrix

Severity decides how loudly the alert arrives — HUD treatment, audio cue, dwell, repeat policy — per the matrix below. Critical is unmissable; medium never interrupts at all. The routing is data, not code: one table the SME can read and sign.

matrix-drivenaudio mirrors visualmedium = log-only
Suppression policy · v0 — same discipline as Phase 1: every constant is a logged knob
ACK (Enter): suppress this tracked hazard for 10 min RE-ARM when: severity escalates OR proximity jumps ≥ 1 zone OR ack expires REPEAT: critical unacked → re-cue audio every 20 s (visual stays) high unacked → one repeat at 60 s, then hold visual only RATE CAP: non-critical alerts ≤ 6 / hour — past the cap, high degrades to log+chime NEVER: suppress a critical class-wide · stack two alerts · scroll anything

The rate cap is the alert budget made mechanical. If the system wants to exceed it, that's a signal the model or the persistence windows need tuning — not that the worker needs more noise.

Anatomy — five rules

  • 01≤ 3 words of hazard, ≤ 7 of action. Read in one glance at arm's-length focus; no sentence ever wraps twice.
  • 02Severity is color + motion, not more words — red pulses, amber sits still. Green/amber/red stay functional, per the brand system.
  • 03One action verb, always. "Step clear." "Don a hard hat." Never a paragraph, never two options.
  • 04Enter acknowledges, Escape dismisses — the same D-pad grammar as Field Coach and Hazard Watch, zero new learning.
  • 05Nothing ever scrolls. If it doesn't fit the card, it doesn't belong on the glass — it belongs in the log.

Trust, measured

the numbers the wear-test must hit

"Do workers trust it" sounds soft; it isn't. Four proxies, logged automatically, reported per session. These are planning targets to calibrate against the first real crews.

≤ 1 / shift
False alarms a worker would call wrong — the metric that predicts muting.
≤ 5 s median
Ack latency — long latency means alerts are being ignored, not read.
0 mute events
Glasses removed or audio disabled mid-task — any occurrence is a design failure to investigate.
≥ 90% agree
Post-task review: alerts the tester agrees were worth an interruption.

The wear-test protocol

Phase 1 tested the pipe; this tests the judgment

Protocol

  • 01Real task as the workload: the tester runs a Field Coach procedure (valve / motor / forklift) start to finish while wearing the system — the distraction is the point.
  • 02Staged hazards around the task: scripted mix across Tier 1 classes — some persistent, some transient, some recurring after acknowledgment.
  • 03Deliberate traps: two hazards at once (arbitration), an acked spill that stays put (suppression), a hazard that escalates (re-arm).
  • 04≥ 5 testers × 2 sessions, counterbalanced hazard scripts; everything logged to the event schema.

Pass bar

  • ✓Task completes — the procedure gets done without removing the glasses.
  • ✓Trust metrics hit — all four targets above, measured from logs.
  • ✓Arbitration correct — in every two-hazard trap, the higher severity showed and the lower logged.
  • ✓Suppression correct — no re-alert on acked static hazards; guaranteed re-alert on escalation.

The event log — Phase 5's seam

Every judgment writes one JSONL event on-device: detection, score, arbitration outcome, alert shown/suppressed, ack, expiry — with the policy version that produced it. No cloud in this phase, but the schema is designed once, here, so the Phase 5 fleet dashboard and the Phase 2 flywheel both read the same stream. Suppressed events matter most: they're the record of what the system chose not to say, which is exactly what a safety review wants to audit.

Policy engine. Tracking + per-class persistence, severity table, arbitration, suppression state machine — as a pure module with the event schema, unit-tested against recorded Phase 1/2 frame logs before it ever touches hardware.

Deliberately out of scope

Still thin — these wait

  • No new detection classes. Judgment works on whatever Phase 2 shipped.
  • No fleet dashboard. The schema ships; the dashboard is Phase 5.
  • No per-site configuration UI. Policy is a reviewed table in the build, not a settings screen.
  • No voice or gesture input. D-pad ack only — matching the platform.
  • No supervisor escalation path. Man-down style auto-escalation designs in Phase 4 with real site contacts.
  • No Android. Same slice platform until Phase 5.
Exit criteriaA tester completes a real procedure wearing the system, gets timely, uncluttered alerts they agree with, the traps all resolve correctly, and the four trust metrics hit — measured from the event log, not impressions.
What Phase 3 ships. The judgment module (drop-in at Phase 1's stage 4) with its versioned, SME-signed policy table; the on-glasses alert component set in the Hazard Watch design language; the JSONL event schema that Phase 5's fleet view and Phase 2's flywheel both consume; wear-test results against the four trust metrics; and the tuned constants memo. After this phase the system doesn't just see hazards — it exercises judgment a worker can live with all shift. That's the difference between a demo and a product, and it's what Phase 4 takes to a real site.

All thresholds (ack window, rate cap, dwell times, trust targets) are planning defaults to calibrate in the wear-test and again at pilot — the structure is the commitment, the numbers are knobs. Severity policy is owned by the safety SME and versioned with the build.
