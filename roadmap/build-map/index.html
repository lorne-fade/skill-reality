Putting real hazard vision
on the glasses

Today's Hazard Watch web app reads motion sensors and simulates the camera view. Making the vision real — the camera actually spotting a missing guard or a spill — means stepping off the web platform and onto Meta's native Device Access Toolkit. Here's the route, the pieces, and the unknowns to pin down first.

The platform shift

The web app and the camera app are two different animals. The web app is instant and needs nothing but a URL; it can never touch the camera. The camera app is a native mobile build that pairs with the glasses — more capable, but review-gated until Meta opens general publishing in 2026.

Today — Web App
Instant URL, sensors only

Loads on the glasses from a link. Motion, orientation, geolocation, local storage. No camera, no mic — a hard platform limit. This is where coach/, hud/ and Hazard Watch live.

Next — Device Access Toolkit
Native app, camera POV

An iOS/Android app paired through the Meta AI app. Hands-free POV camera capture, open-ear audio, display output. Distributed to your own testers now; public listing in 2026.

How it fits together

The detection app does not run on the glasses — it runs on a paired phone, with the Meta AI app as the link layer. Camera frames flow from the glasses to the phone, the phone runs the detection, and concise alerts flow back to the glasses' display and open-ear speakers.

Three subsystems to build

◉ Capture

Pull the POV frames off the glasses through the Toolkit and feed a steady pipeline to the detector. Cadence is the open question — design for periodic frame-grab (a few per second) before assuming full video.

Run a lightweight object detector on each frame, on the phone, so nothing leaves the device. Classes are the safety vocabulary: PPE, spills, missing guards, exposed wiring, blocked exits, person–vehicle proximity.

The build, in phases

~3–4 months to a field pilot
0

Access & de-risk spike

1–2 weeks

Get into the developer preview and answer the questions the docs don't — before writing real product code. This phase exists to kill surprises.

  • Enable Developer Mode; join release channels
  • Stand up the Mock Device Kit + a physical unit
  • Measure camera cadence, resolution, latency
  • Probe what the HUD can actually render
Exit criteriaWe know frames-per-second, glass-to-phone latency, and whether the display can show boxes or only cards. Go / no-go on "real-time."
1

Thin vertical slice

2–3 weeks

One hazard, end to end, on real hardware: prove the whole chain before widening it. Pick "no hard hat" — a well-served, high-value PPE class.

  • Capture → phone → off-the-shelf detector
  • Single alert pushed to the HUD + audio
  • Wire in the Hazard Watch alert UX
  • Instrument latency + battery draw
Exit criteriaWalk up to a bare head, get a hard-hat alert on the glasses within a second or two. The pipeline is real.
2

The detection model

3–5 weeks

Replace the stock detector with a purpose-built one that knows your worksite's hazards, small and fast enough to run on a phone.

  • Assemble + label a hazard dataset
  • Fine-tune a YOLO-n / MobileNet-SSD model
  • Quantize + export to Core ML / TFLite
  • Evaluate recall per class — misses are the risk
Exit criteriaA quantized model hitting agreed recall on a held-out set for the first 4–6 hazard classes, running at target FPS on-device.
3

Hazard logic & on-glasses UX

2–3 weeks

Detections aren't alerts. Add the judgment layer — severity, debouncing, "don't cry wolf" — and the calm glanceable output a worker can trust mid-task.

  • Severity + persistence rules per class
  • Debounce so flicker never nags
  • Visual + audio alert routing
  • On-device log → optional fleet sync
Exit criteriaA tester wears it through a task and gets timely, uncluttered alerts with a low false-alarm rate they'd tolerate.
4

Field pilot

3–4 weeks

Put it on real heads in a real environment through the invite-only tester channel, and tune against ground truth — especially the hazards it misses.

  • Release to a pilot crew via test channel
  • Log detections vs. observed reality
  • Tune thresholds; retrain on hard cases
  • Battery, comfort, and consent review
Exit criteriaA signed-off pilot: measured detection rate, acceptable false-alarms, and a safety sign-off on failure modes.
5

Harden & path to publish

ongoing → 2026 GA

Turn the pilot into a product: fleet dashboard, OTA model updates, privacy posture, and the partner review needed to publish publicly when Meta opens it.

  • Fleet dashboard + audit trail
  • OTA model update channel
  • Privacy / data-handling docs
  • Apply for publish access
Exit criteriaPublishable build, documented privacy stance, and a repeatable path to add hazard classes without a re-architecture.

Validate these first

the Phase 0 spike exists for this

Meta's public docs confirm the Toolkit gives camera, audio, and display access on a paired phone — but they don't publish the numbers that decide whether "real-time hazard vision" is a product or a science project. Pin these down in week one.

UnknownWhy it decides the designHow to de-risk in the spikeSeverity
Camera cadence & latency Continuous video vs. periodic capture changes everything — "real-time" may mean a few frames per second, not 30. Stream from a physical unit and log frame timestamps + glass-to-phone delay. Critical
Display overlay richness If the HUD shows cards/text, not registered boxes, the on-glasses UX is a glanceable alert — the boxes live in review, not live overlay. Push a test overlay and see what the Display API actually accepts. Critical
Battery & thermal Continuous capture + on-device inference is a power draw; a 20-minute battery isn't a shift-long tool. Run the Phase 1 slice for 30–60 min and measure drain + heat. High
Detection recall in the field A missed hazard is a safety failure, not a bug. Field conditions — glare, motion, tiny objects — hurt recall. Evaluate on held-out worksite footage; treat false-negatives as the primary metric. High
Publishing & privacy path Public listing is partner-gated until 2026, and always-on POV camera carries real consent obligations. Confirm tester-channel limits; draft the capture-consent + data-handling stance early. High

What carries over — and who builds it

Reused from today's demos

  • ✓Brand system — palette, type, dark-safe tile logo, the whole visual language ports straight over.
  • ✓Hazard taxonomy & advisory copy — the classes and "here's the one action" wording are already written.
  • ✓Alert UX — the status banner, severity colors and glanceable layout become the HUD language.
  • ✓Sensor monitor ships now — incline / vibration / man-down needs no camera, so it can launch inside the native app on day one.

Team & shape of effort

  • ▸1 native mobile engineer (iOS or Android to start) — Toolkit integration, capture pipeline, HUD output.
  • ▸1 ML engineer — dataset, training, quantization, on-device eval.
  • ▸~0.5 designer — adapt the alert UX to the HUD's real constraints.
  • ▸Safety SME, part-time — hazard classes, severity, and sign-off on failure modes.
Bottom line. A working prototype — one hazard, end to end on real glasses — is roughly a month out, gated only by preview access and the Phase 0 answers. A tuned, safety-reviewed field pilot is a 3–4 month effort. Public distribution waits on Meta's 2026 general availability. The honest first step is the Phase 0 spike: it's cheap, and it tells us whether continuous on-glasses hazard vision is a build or a wait.
